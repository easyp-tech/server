# Phase 12 — Plan 01: Logging Infrastructure

**Requirements:** INFR-01, INFR-02, INFR-03
**Depends on:** Phase 11 (Logging Foundation)
**Estimated tasks:** 4
**Risk:** Low — isolated to `interceptor.go` (new), `api.go`, and `main.go`

## Changes

### 1. INFR-01: Correlation ID Propagation (`main.go`)

- `loggingMiddleware` generates a random 8-char hex `request_id` when `X-Request-Id` header is absent
- Request ID attached to context via `connect.WithRequestID(ctx, id)` for downstream extraction
- - No correlation ID propagation means logs from concurrent requests are indistinguishable

### 2. INFR-02: Connect RPC Logging Interceptor (`interceptor.go` — new)

- `NewLoggingInterceptor(log)` returns a `connect.Option` wrapping a unary interceptor
- Logs at DEBUG:
  - `rpc started` — procedure, peer, request_size
  - `rpc completed` — duration, response_size
  - `rpc failed` at WARN — duration, error, error code
- Uses `connect.RequestIDFrom(ctx)` to include `request_id` in interceptor log lines

### 3. INFR-02: Wire Interceptor (`api.go`)

- `New()` accepts `...connect.HandlerOption` and passes them to all 3 Connect handler constructors
- Change import alias from `connect` → `v1alpha1connect` for generated package to make room for `connectrpc.com/connect`

### 4. INFR-03: Middleware Logging Demotion (`main.go`)

- Error-level middleware logging (WARN for 4xx, ERROR for 5xx) replaced with consistent INFO-level timing/status logging
- Detailed error diagnostics are handled at handler level (interceptor in Phase 12, error-path logging in Phase 13)
